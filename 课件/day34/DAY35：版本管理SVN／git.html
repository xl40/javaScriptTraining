版本管理的概念
    版本管理是软件配置管理的基础，它管理并保护开发者的软件资源。
    
    一个软件从建立，到最后发布上线，如果把每一次提交都算上，你觉得它大概要经历几万个版本？

    版本管理最大的好处就是可以保留我们的历史版本，在代码开发到一半的时候，不至于无故丢失，还可以查看BUG的来龙去脉。那怎样做到这一点呢？如果每一次修改，都保留一个文件副本，那需要多少硬盘才能装的下？

    版本管理种类：集中式的版本管理和分布式的版本管理。实现的控制软件分别有SVN和Git。
    
    SVN是集中式版本管理，大家都提交文件到同一个地方，这服务器的存储压力就大了。因此它对于文件的版本控制，只记录修改部分，不产生多余的数据。


一、SVN介绍
介绍
    SVN是Subversion的简称，是一个开放源代码的集中式的版本控制系统。
    
    集中式代码管理的核心是服务器，所有开发者在开始新一天的工作之前必须从服务器获取代码，然后开发，最后解决冲突，提交。所有的版本信息都放在服务器上。如果脱离了服务器，开发者基本上可以说是无法工作的。

    最大的缺点：这是一个集中式的版本管理，所有的数据都经过中央服务器进行交换同步。如果不能连接到服务器上，基本上不可以工作，如果服务器不能连接上，就不能提交，还原，对比等等。


使用
    1. 安装svn客户端
    2. 你们公司用的svn保存记录代码，拿到 svn地址 和账号密码
    3. 下载代码  checkout 代码
    4. 每天开发新代码之前都需要 先update，然后开发，开完后再 commit提交

二、Git介绍
介绍
    Git是目前世界上最先进的分布式版本控制系统。

Git和SVN的区别
    SVN的存储需要依赖一个服务器，而git所有的东西是放在线上的。节约成本，省时省力。

    git 是分布式的，svn 不是。
    git 按照源数据的方式存储内容，svn 是按照文件的形式存储
    git 和 svn 中的分支不同。
    git 没有全局版本号，svn 有。
    git 内容的完整性优于 svn。

区域划分：
    工作区：个人 PC 中能看到的文件目录结构
    暂存区：stage/index，一般存放在.git/index 中，所以 git 中的暂存区也会称为索引。
    版本库：工作区中的.git 隐藏文件，不算是工作区，而是 Git 的版本库。

四、将文件提交到git版本库
1.在git的工作区内，打开 Git Bash
    输入: git add 文件名
    将文件添加到暂存区等待提交



    命令: git add 文件名
        git add .
        git add *
        git add --all
        git add -A

2.输入: git commit -m "日志信息"
    把文件提交到git的版本库



    命令: git commit -m '提交时候的备注（相当于注释）'

3.输入: git status
    查看commit状态
    出现如下提示，说明commit成功



    命令: git status

    修改下readme.txt里面的内容。并重新使用git status命令 。



    会发现出现了一些变化。git告诉我们，git进行了一些修改，但是这些修改并没有被提交

4.输入: git diff
    查看修改内容
    会出现更改信息



    命令: git diff 你的文件名
        git diff                  工作区和暂存区的差别
        git diff --cached         暂存区和版本库的差别
        git diff HEAD             工作区和版本库的差别

    将更改后的东西提交上去，和之前一样。先add 然后再commit



5.使用: git log
    查看版本修改日志



    命令: git log
        git log --oneline
        git log -n

6.使用: git reset —hard 版本号
    进行版本退回



    命令: git reset —hard 版本号
        git reset --hard HEAD^  退回到上个版本
        如果需要退回好多版本就在后面加上 ^。
        例: git reset --hard HEAD^^  退回两个版本。

    然后去查看readme.txt是否成功退回。

8.将本地git版本库上传到远程仓库
    先与远程仓库建立连接
    再将自己的库push到远程仓库中



    命令: git remote add origin https://github.com/你的账号/你的项目名称

    命令: git push -u origin master 将本地的库推送到master分支 （就是推送到服务器上）

    登录远程仓库，查看是否推送成功。 

    从现在开始，只需要git push就可以将本地的版本库推送到远程仓库

9.使用: git clone 
    从远程仓库下载项目



    命令: git clone https://github.com/liyangyf/test

    将远程仓库的项目克隆下来
    
    克隆之后，如果有权限修改当前项目，可以直接在当前项目内git push即可


10.如果在建立远程连接时输入错误，或提示错误信息：fatal: remote origin already exists.

    解决办法如下：

    1. 先输入：git remote rm origin
    2. 再输入：git remote add origin git@github.com:djqiang/gitdemo.git 就不会报错了！
    3. 如果输入：git remote rm origin 还是报错的话
        error: Could not remove config section ‘remote.origin'
    4. 需要在git的安装路径中，找到一个名为gitconfig的文件，把里面的[remote "origin"]那一行删掉

团队工作场景：
1. 老大A 将 组员B加入开发者权限

2. 组员B，clone代码后立即创建个人新分支

3. 在新分支开发项目，上传分支后，告知老大A

4. 老大A抓取远程分支：git fetch 连接别名 远程分支名

5. 老大A检查代码，处理冲突后，合并分支git merge，并上传主分支

6. 组员B，拉取在线版本库并合并到本地版本库：git pull
  提示：git pull = git fetch + git merge
